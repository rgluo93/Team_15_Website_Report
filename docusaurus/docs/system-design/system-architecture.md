# System Architecture

## Existing setup 

As our main goal is to integrate AI features into the existing LeadNow platform, we had to work with the client’s existing system architecture. Below is the system architecture diagram at the start of our project.

![Existing setup](/img/system-design-initial.png)

Users can access the LeadNow platform either through two interfaces, a website or through a mobile app. Both of these client interfaces connect to a central backend built with Laravel (a PHP backend framework). The web application uses vanilla HTML, CSS, and JavaScript to display UI, with pages rendered on the server using server-side rendering. Server-side rendering is the process of using the Laravel backend to dynamically generate HTML pages which are sent to the user's browser via HTTP. The mobile application is built with Flutter and interacts with the backend through a RESTful API via HTTP.  This is slightly different to the web application because instead of receiving pre-rendered pages, the Flutter app sends API requests to Laravel and then renders the UI on the client side. Laravel also acts as the central data access layer, communicating with a MySQL database to store and retrieve application data, including user information, modules, and other platform content. This is a classic system architecture for application development, employing separation of concerns to enable the system to support both web and mobile platforms effectively.

Our system incorporates multiple components together. As the system architecture is complex, it will be broken down into multiple sections and then finally described as a whole. 

## General entrypoint 
To begin, the diagram below illustrates the general entry point to the AI resource.

![General architecture](/img/system-design-entrypoint.png)

Neither the web application nor the mobile application connect directly to the FastAPI backend. Instead, all requests are routed through the Laravel backend, which acts as an intermediary. Laravel then makes HTTP requests to REST API endpoints exposed by the FastAPI service. To access the GPT-4o-mini large language model, Azure AI Foundry is used. This service, provided by Microsoft Azure, exposes an API that enables requests to be made to a selected LLM hosted in the cloud. All functionalities within the system that require an LLM interact with this API. Moreover, Redis is used to store conversation history for both the general educational assistant and the education officer agent, allowing context to be maintained across messages. In addition, the Vosk model is hosted on the FastAPI backend to provide speech-to-text capabilities. When a user speaks into their microphone, the audio is recorded as a file and sent to the Laravel backend, which forwards it to FastAPI. The FastAPI service then uses Vosk to transcribe the audio into text and returns the result to the frontend.

## RAG pipeline 
Next, our solution utilises retrieval-augmented generation (RAG) to provide contextualised responses. The system architecture for this process is illustrated below.

![Rag system architecture](/img/system-design-rag.png)

Our implementation of RAG is built on top of a vector database. To populate this database, we were provided with PDF documents containing LeadNow resources, such as course content and question-answer pairs from experts. We developed an ETL data pipeline in Python, accessible via a command-line interface (CLI), to process these documents. The pipeline extracts text from the PDFs, splits it into smaller chunks using LangChain’s Recursive Text Splitter, and converts each chunk into vector embeddings using the all-MiniLM-L6-v2 embedding model. These embeddings, along with the corresponding text and metadata, are then stored in the Qdrant vector database. During runtime, the FastAPI backend handles incoming queries which require RAG. Each query is first converted into a vector embedding using the same all-MiniLM-L6-v2 embedding model. A semantic search is then performed on Qdrant to retrieve the most relevant text chunks. These retrieved contexts are incorporated into the prompt, which is subsequently sent to the GPT-4o-mini model via Azure AI Foundry. This process ensures that responses generated by the LLM are grounded in relevant domain-specific knowledge.

## MCP server

We integrated a Model Context Protocol (MCP) server with the FastAPI backend, which acts as an MCP client to provide access to external tools that can be used by the education officer agent. MCP is a standardized protocol that enables LLMs to interact with tools in a structured manner. Communication between FastAPI and the MCP server is carried out over HTTP using the MCP protocol. The system architecture for the MCP server is shown below.

![MCP system architecture](/img/system-design-mcp.png)

The MCP server provides access to three tools. The first tool connects to the LeadNow MySQL database, allowing the agent to execute SQL queries and retrieve information about teachers. For example, if the agent needs to determine how many students have not completed a specific module (e.g. 40.0.0), it generates an appropriate SQL query and sends a tool call to the MCP server, which executes the query and returns the results. The second tool enables communication via WhatsApp using the WhatsApp Cloud API. This allows the agent to send messages to teachers programmatically. For instance, if a reminder needs to be sent to complete a module, the agent extracts the relevant phone number and message content, and sends a tool call to the MCP server, which then forwards the message through the WhatsApp Cloud API. The third tool is a translation service hosted on the MCP server. This allows the agent to translate generated content when required. For example, if the agent generates a set of questions in English but needs them in Swahili, it sends the text to the MCP server via a tool call, where it is processed by the translation model and returned in the target language.

Additionally, education officers can send messages to the LeadNow mobile number via WhatsApp. The WhatsApp Cloud API exposes a webhook that listens for incoming message events to the LeadNow number. When a message is received, the webhook triggers an HTTP request to the FastAPI backend, specifically the education officer agent endpoint. The FastAPI service then processes the request and generates an appropriate response.

## Full System Architecture
Below is the full system architecture diagram 

![Full System Architecture](/img/system-design-full.png)

To host our application, we utilised Azure Container Apps. Using Docker, we built four separate containers: the FastAPI backend (which hosts the speech-to-text and text embedding models), Qdrant, Redis, and the MCP server (which hosts the translation model). Only the FastAPI backend is publicly exposed to the internet, while Qdrant, Redis, and the MCP server are configured to be accessible only within the Azure Container Apps environment. This improves security by limiting direct external access to internal services. This multi-container approach provides better separation of concerns and enables more efficient resource management. Each service can be scaled and configured independently based on its specific workload and resource requirements. For example, the translation model is hosted on the MCP server rather than within the FastAPI backend. This is because the translation model is relatively resource-intensive (≈2 vCPU, 5 GB RAM), whereas the FastAPI backend is lightweight (≈1 vCPU, 3 GB RAM). Since FastAPI handles a higher volume of requests and is deployed with multiple replicas for scalability, embedding the translation model within it would require each replica to provision unnecessary resources, significantly increasing costs. By isolating the translation model in a separate container, it can be provisioned independently with appropriate resources, while the FastAPI service remains lightweight and scalable. This results in a more cost-efficient architecture.
